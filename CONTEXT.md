# Domain Context

The primary artifact produced by this project is a self-contained single-file HTML document.

| Term | Stable definition |
|---|---|
| canonical core | The only editable, host-neutral skill body and shared assets under `skills/vibe-diagram/`. |
| adapter definition | One client's manifest template, output path, additional-file allowlist, and host notes; it contains no copy of the skill body. |
| generated package | A complete client package under `build/<client>/`, composed from the canonical core and an adapter definition. |
| package-static-valid | The builder's production static preflight of the canonical core, adapter, manifest, package contents, paths, and hashes passed. `build-report.json` with `static_validation: passed` expresses only this layer. |
| static-valid | The deterministic builder check, generated package projection comparison, clean diff check, and canonical archive validation passed. Evidence stays in command or CI output rather than tracked documentation. It does not mean that a client was installed or invoked. |
| local-release-ready | The repository-side publication structure, documentation, version metadata, CI definitions, and generated projections satisfy static gates locally, while no public GitHub state or real-client lifecycle evidence is claimed. |
| publication repository | The single repository that owns the canonical core, package definitions, tracked Codex marketplace projection, CI definitions, and release history. |
| public source set | The tracked publication content comprising product source, package and CI definitions, three public policy documents, and generated Codex marketplace artifacts. Task records, design decisions, and validation evidence are not part of it. |
| public publisher identity | The platform-verified individual or business identity selected for a public listing and matched to its name, website, support contact, privacy policy, and terms. Repository ownership alone does not establish this identity. |
| Codex public distribution target | A skills-only plugin approved and published in the public Plugins Directory so users can discover and install it from the Codex App and the Codex CLI plugin browser, with GitHub-path skill installation retained as a direct-install fallback. It does not imply inclusion in the curated `$skill-installer` index. |
| GitHub-path skill installation | Direct installation of a version-pinned canonical skill from the public GitHub repository. Its completed lifecycle covers clean installation, fresh-session discovery and invocation, transactional replacement without a retained predecessor, and uninstall isolation; it does not make the skill searchable in the curated `$skill-installer` index or the public Plugins Directory. |
| transient activation slot | updater 在同级 staging 目录中短暂保存当前安装树的位置，仅用于候选提升失败时在本次进程内恢复；候选成功启用后必须立即删除，不能成为可回滚备份。 |
| verified legacy updater backup | 旧 updater 首次迁入无备份版本时可能留下的目录；只有旧命名格式、`VERSION`、manifest 与完整目录摘要同时匹配才允许由新 updater 清理，手工或已修改目录不在清理范围内。 |
| public availability | Availability in every country or region offered by the OpenAI plugin submission portal, without an additional publisher-defined regional restriction. |
| approved public policy set | The publisher-approved privacy policy, terms of use, and support statement under `docs/public/`, used as the authoritative public URLs in the OpenAI plugin listing. |
| public submission promotion sequence | The release candidate is first exercised through installation, discovery, invocation, HTML delivery, upgrade, and uninstall in the current macOS Codex App and Codex CLI. Only a passing candidate may be promoted to `0.1.0` and submitted as the stable public plugin. |
| stable candidate upgrade path | Install the public `v0.1.0-rc.2` baseline, upgrade it to a locally built `0.1.0` candidate, and publish the `v0.1.0` tag only after the upgraded candidate passes the declared real-client lifecycle. |
| public-release runtime scenario | In both the macOS Codex App and Codex CLI, discover and invoke `vibe-diagram` to generate an evidence-backed, self-contained HTML architecture diagram of the repository's canonical-core-to-public-distribution flow. The artifact must pass the project linter, open without remote resources, survive the declared upgrade, and become undiscoverable after uninstall. |
| independent Codex runtime lanes | The macOS Codex App plugin lane uses a local or repository marketplace, while the Codex CLI skill lane uses the canonical skill from a GitHub repository path. Each lane must independently prove its lifecycle; evidence from one lane cannot satisfy the other. |
| runtime-verified | Installation, discovery, invocation, output, delivery, upgrade, and uninstall were verified in a real client. The conclusion is always scoped to one version and one client lane and cannot be inherited by another release or distribution surface. |
| Codex marketplace projection | The tracked, builder-only generated package under `plugins/vibe-diagram/` used by the repo marketplace. It is not canonical and must match the Codex package and release version at the same stable tag. |
| stable promotion gate | The fail-closed boundary that permits a Codex stable release only after every declared CI, installation-entry, client-surface, and runtime-lifecycle requirement has passed; any failure keeps the version at release-candidate status. |
| direct stable release | A maintainer-authorized path that requires local `static-valid` evidence, then immediately publishes the matching `main` commit, immutable tag, GitHub Release, and moving `stable` branch without waiting for GitHub Actions. CI remains asynchronous evidence, and this path never implies `runtime-verified`. |
| sequence interaction kernel | The same-version CSS/JavaScript enhancement embedded in six sequence templates. It handles only width, zoom, scrolling, sticky headers, and degradation; it does not generate or rewrite business semantics. |
| semantic participant | An independent sequence participant with a stable `data-participant-id`; semantically different participants must not be merged merely to reduce columns. |
| primary sequence message | A message node declaring structured `data-from`, `data-to`, and `data-message-kind`. Calls, returns, async messages, self-calls, and exceptions count; notes, evidence annotations, and phase headings do not. |
| major sequence phase | A main business phase explicitly declaring `data-sequence-phase-id`. Loops, branches, and notes count only when they also declare this attribute. |
| overview sequence | The main-path overview for an over-complex diagram. It keeps phase boundaries and critical transitions and does not pretend to be the complete call chain. |
| detail sequence | A diagram mapped to one overview phase that preserves every real participant, message, and exception branch for that phase. |
| adaptive viewport contract | A host-neutral presentation contract that declares a diagram canvas's width, height, scrolling, zoom, sticky-navigation, mobile, print, no-JavaScript, and reduced-motion behavior without defining its business grammar. |
| semantic relation contract | A static structural contract that gives canvases, semantic objects, groups, and directed relations stable identifiers so readability, endpoint integrity, complexity, overview/detail mapping, and semantic fallbacks can be validated without parsing visible labels or CSS classes. |
| global artifact shell | The versioned presentation shell shared by all 59 templates: title and conclusion first, one compact reading guide second, the interaction hint directly above zoom controls on the guide's right, and the primary canvas third. Its DOM order, evidence states, guide groups, controls, responsive wrapping, print behavior, and embedded kernels are fail-closed build inputs. |
| content-neutral layout template | A canonical template that fixes only topology, relative position, hierarchy, geometry, connection anchors, complexity budget, responsive transformation, and interaction capability. Every visible canvas title, icon, node, relation label, description, detail, fallback, evidence claim, semantic role, and relation kind comes from the current artifact; neutral `layout-slot-NNN`, `layout-node-NNN`, `layout-detail-NNN`, `layout-relation-NNN`, and `canvas-*` placeholders carry no domain meaning. |
| unfilled template preview | The browser rendering of a canonical template before task content is substituted. It suppresses unresolved `canvas-text` tokens across every canvas while preserving their source slots, geometry, references, and the generated artifact's real labels. |
| computed layout audit | 全部模板共享的无截图浏览器自检：在字体就绪和尺寸变化后读取真实 DOM/SVG 几何，检查节点重叠、内容溢出、关系线穿越节点或文字、边界锚定、画布利用率、阅读引导与缩放控件位置以及页面横向溢出，并通过 `data-computed-layout-audit` 输出机器可读结果。它不生成业务语义，也不等同于真实客户端生命周期验证。 |
| diagram family policy | A trusted canonical policy for one diagram family. It declares allowed semantic profiles, relation kinds, complexity budgets, required reading aids, and fallback strategies; a template cannot override its own policy or raise its own budget. |
| semantic profile | A per-canvas structural vocabulary such as graph, matrix, timeline, artboard, or ledger. A template may contain more than one profile while retaining its own family-specific visual grammar. |
| semantic reading guide | An author-provided, structurally linked navigation aid such as a sticky lane label, stage header, axis header, boundary index, or landmark index. The shared runtime may position it but must not infer or generate its meaning. |
| semantic mobile fallback | A baseline HTML representation that preserves identities, relation endpoints, direction, ownership, ordering, axes, boundaries, and evidence when a desktop visual topology cannot remain readable on a narrow viewport. It is not an unrelated summary or a flattened card list. |
| family complexity budget | A fail-closed limit defined by the diagram family policy over semantic objects, relations, groups, depth, rows, columns, views, stages, or other profile-specific dimensions. Exceeding it requires an explicitly linked overview/detail split. |
| progressive disclosure capability | An optional runtime enhancement bound to the semantic relation contract. It opens author-provided detail panels and preserves focus, anchors, print visibility, and no-JavaScript readability; it never creates semantic content. |
| Vibego downstream projection | A controlled copy mapped into Vibego after canonical verification. Host delivery wording, language, and template-relative paths may differ, while the sequence-contract version, kernel digest, template identity, semantic endpoint rules, and structural signatures must match. |
| template-conformant artifact | A generated HTML artifact whose declared canonical template, locked style and script blocks, visual shell, slot inventory, and permitted DOM structure are recomputed and verified rather than trusted from self-declared identity attributes. |
| primary-canvas budget | A presentation limit below the semantic family ceiling. It keeps the first visible canvas focused on the conclusion, primary path, and critical exceptions while moving complete evidence and implementation detail into mapped native details. |
| invocation-complete | A visual request lifecycle that completed update checking, workflow loading, template selection, canonical scaffolding, strict linting, and HTML delivery. Merely loading the Skill or returning Mermaid is not invocation-complete. |
| Vibe Diagram delivery system | The end-to-end product boundary spanning canonical authoring, generated client packages, public distribution, and client-scoped runtime verification. It is broader than the source repository alone. |
| primary canvas direction | The dominant visual reading axis authored into a diagram and proven jointly by semantic ranks, named regions, and visible directed relations. A template name, filename, or prose claim is not direction evidence. |
| side-rail center-spine layout | A landscape geometry with a narrow west rail, a wide center spine read north to south, a narrow east rail, a lower-center band, and a full-width southern band. These positions do not imply entry, runtime, operations, data, or infrastructure semantics. |
| north-to-south layered DAG | A top-to-bottom dependency graph whose ranks progress from north to south, may place peers in parallel within a rank, and may branch or merge between ranks without turning the primary path into a west-to-east chain. |
| authored output language | The language inferred from the user's current request and used for every visible artifact label, including template-authored node titles, regions, legends, controls, details, fallbacks, and evidence wording. Canonical English is a source default, not permission to leak English into a Chinese artifact. |
| node detail disclosure | 一个外层语义节点拥有一个原生主详情入口；节点内部每个独立小节点拥有自己的原生辅助详情入口。所有入口与可打印 `<details>` 一一对应，既避免嵌套链接，也不以图标或弹窗替代节点的主标题和摘要。 |
| node-detail interaction hint | One concise, language-matched instruction inside the shared reading guide that tells readers detailed nodes are interactive. It appears exactly once and never floats inside the primary SVG canvas. |
| evidence boundary banner | A compact, visible evidence ledger inside the global reading guide. It always appears after the title and before the first primary canvas and distinguishes observed implementation, completed checks, and not-yet-verified claims. |
